16""" Provides various functions which together can be used to take a graph and\ 

17 split it into pieces that can be loaded on to a machine, along with\ 

18 routes between the pieces. 

19 

20 Functional Requirements 

21 ======================= 

22 

23 * Creation of an Application Graph of Vertices indicating points of\ 

24 computation within the graph and Edges between the vertices\ 

25 indicating a directional communication between the vertices;\ 

26 and a similar Machine Graph. 

27 

28 * Vertices in the Application Graph will have a number of *atoms*\ 

29 - an atom cannot be broken down in to anything smaller. 

30 

31 * Vertices in the Application Graph must be able to indicate what\ 

32 machine resources are required by any given subset of the atoms. 

33 

34 * Vertices in the Machine Graph must be able to fit on a single\ 

35 chip of the machine in terms of resource usage. 

36 

37 * A Vertex can have a number of constraints which must be\ 

38 respected by any algorithm which uses the graph. Algorithms\ 

39 must check that they can support the given constraints and must\ 

40 fail if they cannot. Provided constraints include support for: 

41 

42 * The maximum number of atoms which any Machine Graph Vertex\ 

43 can contain for a given Application Graph vertex 

44 

45 * The chip and/or processor on to which a Machine Graph\ 

46 Vertex should be placed. 

47 

48 * A set of Application Graph Vertices whose corresponding\ 

49 Machine Graph vertices should contain the same number of\ 

50 atoms. 

51 

52 * A set of Application Graph Vertices whose corresponding\ 

53 Machine Graph vertices should be placed on the same chip\ 

54 if they contain the same atom. 

55 

56 * It should be possible to create new constraints as the need\ 

57 arises. 

58 

59 * Multiple edges can exist between the same two vertices. 

60 

61 * It must be possible to build the Machine Graph directly without\ 

62 requiring that it is created by one of the other modules. 

63 

64 * It is *not* required that there is a Machine Graph Edge between\ 

65 every pair of Machine Graph Vertex from the same Application\ 

66 Graph Vertex. 

67 

68 * Where a Machine Graph is created from an Application Graph, it\ 

69 should be possible to find the corresponding Vertices and Edges\ 

70 from one graph to the other. 

71 

72 * Creation of multicast routing info consisting of key/mask\ 

73 combinations assigned to Edges of the Machine Graph. 

74 

75 * It must be possible to build this information directly without\ 

76 requiring that it is created by one of the other modules. 

77 

78 * There should be exactly one key/mask combination for each\ 

79 Edge in the Machine Graph, which will represent all the keys\ 

80 which will be sent in all packets from the Vertex at the\ 

81 start of the Edge down that Edge. 

82 

83 * It is possible for a Vertex to send several different keys\ 

84 down several different Edges, but only one per Edge (but\ 

85 note that it is acceptable for different keys to be assigned to\ 

86 different Edges between the same two Vertices). 

87 

88 * There should be no overlap between the key/mask combinations\ 

89 of Edges which come from different Vertices i.e. no\ 

90 two Edges which start at different Vertices should have\ 

91 the same key/mask combination. 

92 

93 * Partitioning of an Application graph with respect to a machine, such\ 

94 that the resources consumed by each Vertex does not exceed those\ 

95 provided by each chip on the machine. 

96 

97 * It should be possible to select from a range of partitioning\ 

98 algorithms or provide one, although a default should be provided\ 

99 in the absence of such a choice . 

100 

101 * Any partitioning constraints should be met; if there are any\ 

102 that cannot, or that are not understood by the algorithm in use\ 

103 an exception should be thrown. Non-partitioning constraints can\ 

104 be ignored, although these can be used if it makes sense for\ 

105 the given algorithm. 

106 

107 * It must be possible to create at least one grouping of the\ 

108 generated Vertices so that each group fits within the\ 

109 resources provided by a single chip on the machine. 

110 

111 * It should not be assumed that a given grouping of Vertices\ 

112 will be the final grouping on the machine, although it is\ 

113 acceptable to make hints through additional constraints about\ 

114 what is likely to work. 

115 

116 * The machine itself must not be altered by the partitioning, so\ 

117 that it can be used in further processing. 

118 

119 * The graph itself must not be altered by the partitioning, so\ 

120 that it can be used in further processing. 

121 

122 * No two Machine Graph Vertices created from a single Application\ 

123 Graph Vertex can contain the same atom. 

124 

125 * Any Edges in the Application Graph must be split with the\ 

126 Vertices to create a number of Machine Graph edges, such that\ 

127 where there was a vertex *v* connected to a vertex *w* by a\ 

128 single edge in the Application Graph, there should be\ 

129 an Edge in the Machine Graph between every Vertex of\ 

130 Application Graph Vertex *v* and every Vertex of Application\ 

131 Graph Vertex *w*; for example, if there are 2\ 

132 Machine Graph Vertices for each of *v* and *w*, and one Edge\ 

133 between them in the Application Graph, then there will be 4 \ 

134 new Edges in the Machine Graph for this Edge. 

135 

136 * Placement of a Machine Graph on a given machine, such that the\ 

137 resources required by any combination of Vertices placed on any\ 

138 chip in the machine does not exceed the resources provided by that\ 

139 chip. 

140 

141 * It should be possible to choose from a range of placement\ 

142 algorithms or provide one, although a default should be provided\ 

143 in the absence of such a choice. 

144 

145 * Any placement constraints should be met; if there are any that\ 

146 cannot, or that are not understood by placement algorithm, an\ 

147 exception should be thrown. Non-placement constraints can be\ 

148 ignored, although these can be used if it makes sense for the\ 

149 given algorithm. 

150 

151 * The machine itself should not be altered by placement so that\ 

152 it can be used in further processing. 

153 

154 * The graph itself should not be altered by placement so that\ 

155 it can be used in further processing. 

156 

157 * The returned placements should only contain a single placement\ 

158 for each vertex. 

159 

160 * The placements should be such that the vertices with\ 

161 edges between them must be able to communicate with each\ 

162 other. 

163 

164 * Allocation of multicast routing keys and masks to a Machine Graph\ 

165 such that each vertex sends out packets with a different key/mask\ 

166 combination. 

167 

168 * This can use the placement information if required. If an\ 

169 algorithm requires placement information but none is provided\ 

170 an exception is thrown. 

171 

172 * Routing of edges between vertices with a given allocation of\ 

173 routing keys and masks with respect to a given machine. 

174 

175 * It should be possible to choose from a range of routing\ 

176 algorithms, or provide one, although a default should be\ 

177 provided in the absence of such a choice 

178 

179 * For any vertex, following the routes from the placement of\ 

180 the vertex should result exactly in the set of placements of\ 

181 the destination vertices described by all the edges\ 

182 which start at that vertex. No additional destination should\ 

183 be reached, and no fewer than this set of destinations should be\ 

184 reached. 

185 

186 * It should be possible to call each of the modules independently.\ 

187 There should be no assumption that one of the other modules has\ 

188 produced the data input for any other module. 

189 

190 * There should be no assumption about how the inputs and outputs\ 

191 are stored. 

192 

193 * Any utility functions that provide access to internal structures\ 

194 within a data structure should operate in approximately O(1) time;\ 

195 for example, where an object of type *obj* holds a number of objects\ 

196 of type *subobj* with property *prop*, requesting a list of *subobj*\ 

197 objects contained within *obj* with property value *prop* = *value*\ 

198 should not iterate through a list of such objects, but should\ 

199 instead maintain a mapping that allows access to such objects in\ 

200 O(1) time. If this is not possible, *obj* should only provide access\ 

201 to a list of *subobj* objects, allowing the caller to filter these\ 

202 themselves. This will ensure that no misunderstanding can be made\ 

203 about the speed of operation of a function. 

204 

205"""
